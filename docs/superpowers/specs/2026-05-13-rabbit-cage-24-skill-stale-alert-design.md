# RABBIT-CAGE-24: Fix skill-stale alert — design spec

**Date:** 2026-05-13  
**Bug:** RABBIT-CAGE-24  
**Feature:** rabbit-cage  
**Branch:** backlog/rabbit-cage-24-fix-skill-stale-alert  

---

## Background and research finding

RABBIT-CAGE-24 was filed because `.rabbit-plugins-stale` — written by `build.sh` when skill artifacts are rebuilt — persisted for the entire session, causing `sync-check.sh` to emit a `/reload-plugins` alert on every Stop even after the user ran the command.

Root cause investigation revealed the alert was prescribing the wrong action entirely.

**Official Claude Code documentation states (verbatim):**

> "Claude Code watches skill directories for file changes. Adding, editing, or removing a skill under `~/.claude/skills/`, the project `.claude/skills/`, or a `.claude/skills/` inside an `--add-dir` directory **takes effect within the current session without restarting**."

> "When you or Claude invoke a skill, the rendered `SKILL.md` content enters the conversation as a single message and **stays there for the rest of the session**. Claude Code does not re-read the skill file on later turns."

**Implication:** For local project skills in `.claude/skills/`, `/reload-plugins` is irrelevant. The file watcher picks up changes automatically. The only case requiring user action is when a skill was already invoked earlier in the same session — in that case, re-invoking the skill (not `/reload-plugins`) picks up the new content.

The entire `rabbit-plugins-stale` mechanism was solving a non-problem and prescribing the wrong remedy.

---

## Decision

Remove the stale marker mechanism. Replace with a one-time informational notification that names which skills were rebuilt and states the correct behavior (auto-refresh on next invocation).

---

## Design

### Files changed

| File | Change |
|------|--------|
| `.claude/features/rabbit-cage/scripts/build.sh` | Remove stale marker write; write `.rabbit-skills-updated` with skill name instead |
| `.claude/features/rabbit-cage/hooks/sync-check.sh` | Remove stale marker check; add one-time `.rabbit-skills-updated` notification block |
| `.claude/features/rabbit-cage/hooks/session-init.sh` | Remove `rm -f .rabbit-plugins-stale` line |
| `.gitignore` | Swap `.rabbit-plugins-stale` for `.rabbit-skills-updated` |

(Deployed artifacts in `.claude/hooks/` and `.claude/scripts/` are regenerated by `build.sh` — not listed separately.)

---

### New marker: `.rabbit-skills-updated`

**Written by:** `build.sh`  
**Format:** Newline-separated skill names (one per rebuilt SKILL.md destination)  
**Example content:**
```
rabbit-bug
rabbit-cage
```

**Cleared by:** `sync-check.sh` on first read (self-clearing — fires once per build, never again)  
**Gitignored:** yes

---

### `build.sh` change

When a `copy-file` target writes to a destination matching `.claude/skills/*/SKILL.md`, append the skill directory name (the `*` component) to `.rabbit-skills-updated`.

Replaces the current block that writes `.rabbit-plugins-stale`.

---

### `sync-check.sh` change

Replace the final plugin-stale block with:

```
if [ -f "$REPO_ROOT/.rabbit-skills-updated" ] && [ "$_json_emitted" -eq 0 ]; then
  # Read updated skill names, emit one-time green notification, delete marker
  names=$(paste -sd, "$REPO_ROOT/.rabbit-skills-updated")
  rm -f "$REPO_ROOT/.rabbit-skills-updated"
  python3 -c "
import json, sys
names = sys.argv[1]
print(json.dumps({'systemMessage': '\x1b[32m[rabbit] Skills updated: ' + names + ' — will reload automatically on next invocation.\x1b[0m'}))
" "$names"
fi
```

Marker is deleted before emitting JSON so a crash after deletion still consumed the marker. Notification fires at most once per build.

---

### `session-init.sh` change

Remove the line:
```bash
rm -f "$REPO_ROOT/.rabbit-plugins-stale"
```

No replacement needed. `.rabbit-skills-updated` self-clears via `sync-check.sh`.

---

### `.gitignore` change

Remove: `.rabbit-plugins-stale`  
Add: `.rabbit-skills-updated`

---

## Success criteria

1. `build.sh` writes `.rabbit-skills-updated` (not `.rabbit-plugins-stale`) after rebuilding a skill
2. `sync-check.sh` emits the notification exactly once after a build, then deletes the marker
3. No further notifications fire on subsequent Stops in the same session
4. `sync-check.sh` emits nothing if no skill was rebuilt
5. The notification message names the specific skills and states "reload automatically on next invocation"
6. `.rabbit-plugins-stale` references removed from all files

---

## Out of scope

- Hooks (`.claude/hooks/*.sh` script files): the docs do not explicitly state whether hook scripts are re-read fresh or cached per invocation. This is a separate investigation item.
- Marketplace plugin skills (superpowers): these live in `~/.claude/plugins/cache/` and may have different caching behavior. Not affected by this change.
