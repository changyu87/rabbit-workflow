════════════════════════════════════════════════════════════════════════
STEP 1 — LOCK
════════════════════════════════════════════════════════════════════════

Set the scope marker as your FIRST write action — and ONLY that:
  touch {{scope_marker_path}}

Do NOT register a `trap '... rm -f ...' EXIT` here. Each Claude Code Bash
tool invocation runs in a separate (per-call) shell process. The trap would
fire the moment that shell exits — i.e., immediately after `touch` returns —
deleting the scope marker before any subsequent step (TEST-WRITE, IMPLEMENT,
etc.) can rely on it. Cleanup is explicit and happens in STEP 8 UNLOCK
(`rm -f {{scope_marker_path}}`), AFTER the chore
commit and BEFORE the HANDOFF block.

════════════════════════════════════════════════════════════════════════
STEP 2 — TEST-WRITE
════════════════════════════════════════════════════════════════════════

1. Read all existing tests in {{feature_dir}}/test/
2. Compare each spec behaviour against existing tests.
3. For each behaviour with no e2e test: add one now. (E2E TEST RULE applies.)
4. Commit new/updated tests:
   git add {{feature_dir}}/test/
   git commit -m "test({{feature_name}}): add e2e tests for spec behaviours"

Note on ordering: the commit above happens BEFORE STEP 3 (TEST-RED) runs.
That is intentional — STEP 3 verifies the suite is failing as expected after
the commit. The tests are committed in their failing state so the diff is
captured atomically alongside the implementation that will turn them green.

STEP 3 — TEST-RED
════════════════════════════════════════════════════════════════════════

Run: python3 {{feature_dir}}/test/run.py
Verify tests FAIL. If they already pass (no implementation gap), document why and proceed.

Bring tdd_state into this cycle from the prior cycle's `test-green` endpoint.
The tdd-step.py state machine is forward-only; from `test-green` the only
valid path into a new cycle starts with `spec-update`. Run this BEFORE the
`test-red` transition below. If the starting state is already `spec-update`
or further along, the transition is a no-op-friendly check via `show` first:

  CURRENT_STATE=$(python3 {{tdd_step_py}} show {{feature_dir}})
  if [ "$CURRENT_STATE" = "test-green" ]; then
    python3 {{tdd_step_py}} transition {{feature_dir}} spec-update
  fi

  python3 {{tdd_step_py}} transition {{feature_dir}} test-red

════════════════════════════════════════════════════════════════════════
STEP 5 — SYNC-DEPLOYED
════════════════════════════════════════════════════════════════════════

This step bridges feature-local source edits to their deployed copies under
`.claude/hooks/`, `.claude/skills/`, `.claude/commands/`, `.claude/agents/`,
and `.claude/settings.json`. Feature-local edits to a source file (e.g.
`hooks/foo.py`) do NOT automatically update the deployed copy; without this
step the committed deployed bytes drift from the source bytes and the next
TDD cycle in any feature observes a manifest-drift FAIL.

1. Read `{{feature_dir}}/feature.json` and enumerate every entry in the
   `manifest` array whose `api` is one of:
     - `publish_file`
     - `publish_hook`
     - `publish_skill`
     - `publish_settings`
   (`publish_generated` runs via `check_drift_regenerate` at runtime and is
   NOT re-published here; `publish_command` and `publish_agent` ARE in scope
   when they appear in the manifest.)

2. For each in-scope manifest entry, invoke the matching API from
   `contract.lib.publish` in-process (lazy-import the module from
   `{{repo_root}}/.claude/features/contract/lib/publish.py`), passing the
   entry's `args` dict plus the keyword-only context params
   `feature_dir={{feature_dir}}` and `repo_root={{repo_root}}`. Each call
   returns a `CheckResult`; collect `passed`/`messages` for every entry.

3. `git add` every deployed path the publish APIs modified. The feature-local
   source changes are already staged from STEP 4; this step stages the
   deployed copies on top of that.

4. At the END of STEP 5, perform the SINGLE atomic commit covering BOTH the
   STEP 4 source edits AND the STEP 5 deployed-copy sync:
     git commit -m "fix({{feature_name}}): <one-line summary>"
   (use `feat(...)` instead of `fix(...)` when introducing a new feature
   rather than fixing a bug)

5. Immediately after the commit, advance state:
     python3 {{tdd_step_py}} transition {{feature_dir}} sync-deployed

On any publish-API failure (a `CheckResult` with `passed=False`), STOP
immediately. Do NOT commit. Do NOT proceed to CODE-REVIEW. Emit this
fail-HANDOFF and stop:

  HANDOFF:
    feature: {{feature_name}}
    tdd_state: impl
    test_result: not_run
    spec_compliance: fail
    tdd_report_path: null
    notes: STEP 5 SYNC-DEPLOYED failed at publish_<api> for <entry>: <messages>

  HANDOFF_JSON:
  ```json
  {
    "handoff_schema_version": "1.1.0",
    "feature": "{{feature_name}}",
    "tdd_state": "impl",
    "test_result": "not_run",
    "spec_compliance": "fail",
    "tdd_report_path": null,
    "closed_items": [],
    "notes": "STEP 5 SYNC-DEPLOYED failed at publish_<api> for <entry>: <messages>",
    "discovered_issues": [],
    "aborted_reason": null
  }
  ```

If the feature's `manifest` array is empty or absent, STEP 5 is a no-op for
the publish loop — but you MUST still perform the commit (step 4 above) to
land the STEP 4 source edits, and MUST still advance state to `sync-deployed`
(step 5 above). The state machine is monotonic; skipping the transition
would wedge the next cycle.
