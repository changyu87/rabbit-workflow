Stable numbers protect cross-references in code, tests, and other features' specs from cascading rewrites; gaps are permitted by `check_invariant_monotonic_order` (strictly increasing, not contiguous).

10. `check-naming.py` documents that the `rbt-` prefix is fully deprecated with no remaining valid use cases; comments and flag messages in that script must not reference `rbt-` as a valid or recommended prefix. The current naming policy is: user-facing artifacts use `rabbit-`; the `rbt-` prefix is banned.
11. Boolean CLI flag values and subcommand values across the rabbit workflow use the literal strings `true` and `false` exclusively. The values `enabled`, `disabled`, `on`, `off`, `yes`, `no` are prohibited as boolean values (action verbs like `lock`, `unlock`, `add`, `remove` remain allowed when the subcommand itself denotes an action, not a boolean state).
12. CLI flag names, subcommand names, and configuration variable names in the rabbit workflow MUST be positive-streamlined: they describe what is present/active, never what is absent/disabled. Names beginning with `no-`, `disable-`, `skip-`, `without-`, or any negating prefix are prohibited. If such a name exists, it must be renamed to describe what IS active when the flag/variable is true. Boolean state is encoded in the value (`true`/`false`), never in the name.
13. `check-tests-non-interactive.py` MUST scan Python test files (`.py`) under `<feature-dir>/test/`, not shell scripts (`.sh`). The repo is Python-only (rabbit-cage Inv 17, rabbit-file Tech Stack); a `.sh`-only scanner is silently vacuous on every invocation. The script MUST detect Python interactive constructs that would block an end-to-end run: bare `input(` calls, `getpass.getpass(`, `click.prompt(`, `click.confirm(`, and any `sys.stdin.read*()` call that is not preceded by an `isatty()` guard or a piped-input fixture. A violation MUST exit 1 with stderr naming the file and the offending construct.

The Python-only stack means `test/run.sh` does not exist; checking for it is silently vacuous and rejects valid features. References to `.sh` test runners are banned across all contract enforcement scripts.

34. Every Python script under `.claude/features/contract/scripts/` (excluding `enforcement/`) MUST have at least one production caller outside the contract feature itself. Deprecation criterion for the regression test: when an automated dead-code detector spanning the whole repo is wired into the Stop hook.
36. Retirement semantics for features: `feature.json` MAY carry a top-level `status` field with enum values `"active"` (default when omitted) or `"retired"`. A retired feature signals that its surface has been absorbed into a successor feature; the directory remains on disk as a tombstone but the feature is no longer actively developed.
    (a) `feature.json.schema.json` MUST declare the `status` field with `enum: ["active", "retired"]` and document the default as `"active"`.
    (b) `validate-feature.py` MUST short-circuit (exit 0 with a `RETIRED:` notice) when `feature.json` has `status: retired`. Retired features are exempt from the standard structural checks (spec.md, contract.md, test/run.py, deprecation_criterion) because their successor owns the live surface.
    (c) `.claude/features/contract/workspace-structure.json` MUST mark retired features (whose tombstone directories remain on disk) as `required: false` so future declaration-vs-disk audits do not flag tombstone directories as missing. Features that have been fully retired (directory removed entirely) MUST NOT appear in `workspace-structure.json` at all.
37. The CLI scripts in `scripts/enforcement/` and `scripts/validate-feature.py` MUST be thin wrappers (~10 lines) that import from `contract.lib.checks`, call the matching function, print the returned messages, and exit 0 / 1.
    (a) The library MUST export a `CheckResult` dataclass with exactly two fields: `passed: bool` and `messages: list[str]`. Each library function returns a `CheckResult`. Messages are human-readable lines (one per issue) that the CLI shim prints to stderr on failure or stdout on success.
    (b) Required function names and signatures: `check_tests_non_interactive(feature_dir: str) -> CheckResult`, `check_sentinel(path: str) -> CheckResult`, `check_naming(root: str) -> CheckResult`, `check_imports_resolve(feature_dir: str) -> CheckResult`, `check_symlinks_resolve(root: str) -> CheckResult`, `check_template_producer_consistency(template_path: str) -> CheckResult`, `check_numbered_lists(targets: list[str]) -> CheckResult`, `validate_feature(feature_dir: str) -> CheckResult`, `check_prompts_section(features_root: str) -> CheckResult`, `validate_prompts_section(feature_dir: str) -> CheckResult`.
    (c) `lib/__init__.py` MUST exist so the package is importable (empty file is sufficient).
    (d) Library functions MUST NOT call `sys.exit`, MUST NOT print to stdout / stderr, and MUST NOT raise on contract-violation conditions — all outcomes flow through the returned `CheckResult`. Invocation errors (missing file, bad JSON, etc.) are reflected as `passed=False` with descriptive messages.
    (e) CLI consumers of the existing scripts (notably `tdd-step.py`) continue to call the CLI shims with their existing argv shape; the refactor is backward-compatible at the CLI boundary. Rewiring those consumers to import the library directly is out of scope for this invariant.

38. **Cross-feature invariant-monotonic-order enforcement.** Contract MUST expose `check_invariant_monotonic_order(feature_dirs: list[str]) -> CheckResult` in `lib/checks.py`, a CLI shim at `.claude/features/contract/scripts/enforcement/check-invariant-monotonic-order.py`, and a regression test `test/test-check-invariant-monotonic-order.py` wired into `test/run.py`. The check parses each feature's `docs/spec/spec.md`, walks its `## Invariants` / `### Invariants` sections, extracts top-level numbered items (`^(\d+)\.`), and asserts numbers appear in strictly increasing order WITHIN each section (gaps are permitted — they record retired-invariant tombstones; only repeats and decreases fail). A `KNOWN_ISSUES` allowlist inside the library function permits features pending a renumber to be skipped from enforcement; the current value is `[]` (all features pass the strict-increase check including those with deliberate gaps). Operators MUST add an entry only when a renumber cycle is in flight, and MUST remove it once the renumber lands. Features absent from `KNOWN_ISSUES` are validated; any monotonicity violation FAILs the test with a clear `<feature>:<section>: <prev_num> → <next_num>` diagnostic. Features added later are validated automatically (no per-feature opt-in needed).

39. **Retired-invariant log.** `.claude/features/contract/CHANGELOG.md` MUST exist, carry YAML frontmatter (`feature: contract`, `owner: rabbit-workflow team`, `deprecation_criterion`), and serve as the tombstone log for invariants previously declared in `docs/spec/spec.md` and since retired. Each tombstone entry MUST name the original invariant number (as it appeared in spec.md at the time of retirement), a one-line "what it asserted + why retired" summary, and the backlog/wave ID that drove the retirement. Tombstone numbers in CHANGELOG.md record historical numbers and are NOT updated when spec.md is renumbered. The CHANGELOG MUST also carry a top-of-file note for each significant renumber event naming the cycle that performed it, OR — when gaps are preserved rather than renumbered — a note documenting that the gaps are intentional and which numbers they cover.

44. **`contract.lib.publish` API library.** `.claude/features/contract/lib/publish.py` MUST export seven publish API functions: `publish_file`, `publish_skill`, `publish_command`, `publish_agent`, `publish_hook`, `publish_settings`, `publish_generated`. Each accepts API args as explicit params plus keyword-only context params `feature_dir` and `repo_root`, and returns `CheckResult` from `lib.checks`. All functions are idempotent: re-running with unchanged source is a no-op (SHA-256 comparison for file-copy operations; content-equality check for generated content). `publish_hook` additionally reads/writes `.claude/settings.json` via read-modify-write, preserving all existing fields, and is idempotent on the hook command registration. `publish_settings` MUST be merge-aware on `.claude/settings.json` rather than a raw file copy, because the destination `hooks` section is the shared registration surface for every feature's `publish_hook` calls — overwriting it clobbers cross-feature hook registrations and prevents the drift-detection loop from ever reaching a stable state. The merge rule is: read the existing `.claude/settings.json` (treat missing/invalid/non-object content as `{}`); overlay every non-`hooks` field from the source (source-wins for non-hooks fields); for the `hooks` section, start from the existing event keys and, for each event present in the source, append source hook entries whose `command` strings are not already present in the destination event's flattened command set (matched by exact command string equality, the same identity used by `publish_hook`). Existing hook entries from other features MUST be preserved. The call is a no-op (no write, message contains "no-op") when the merged result equals the existing on-disk content. Coverage enforced by `test/test-publish-{file,skill-command-agent,hook,settings,generated}.py`; `test/test-publish-settings.py` MUST additionally exercise: (a) a pre-existing hook from another feature is preserved when a source with a different hook is published, (b) re-publishing the same source twice against a destination that already holds another feature's hook produces no duplicate entries, (c) non-`hooks` fields in the source replace those in the destination, (d) a destination with no `hooks` key accepts source hooks cleanly, and (e) a source with no `hooks` key leaves existing destination hook registrations untouched.

45. **`contract.lib.mutation` API library.** `.claude/features/contract/lib/mutation.py` MUST export seven mutation API functions: `write_marker`, `delete_marker`, `set_json_key`, `delete_json_key`, `append_json_array`, `remove_json_array_value`, `run_feature_script`. JSON-path mutators (`set_json_key`, `delete_json_key`, `append_json_array`, `remove_json_array_value`) accept dotted key paths and operate via shared helpers (`_load_json_or_empty`, `_write_json`, `_set_nested`, `_get_nested`, `_delete_nested`) defined once at module top. Marker mutators (`write_marker`, `delete_marker`) operate on filesystem markers and are idempotent (no-op on re-write of identical content; no-op on delete of absent file). Array mutators are idempotent on duplicates (`append`) and on absence (`remove`). `run_feature_script` is the escape hatch — invokes a feature-owned script via subprocess for mutations outside standard primitives (e.g. chmod). All functions return `CheckResult` from `lib.checks`. Hand-rolled, stdlib only. Coverage enforced by `test/test-mutation-{write-marker,delete-marker,set-json-key,delete-json-key,append-json-array,remove-json-array-value,run-feature-script}.py`.

46. **`contract.lib.producers` content-producer library.** `.claude/features/contract/lib/producers.py` MUST export a `call_producer(name, args)` dispatcher and at least three named producers: `read-file`, `expand-at-imports`, `generate-claude-md`. Each producer is a pure function taking named args, returning content as a string. `read-file(path)` is a raw file read. `expand-at-imports(file)` reads a file and expands every `@<path>` line (one level only, non-recursive — matches Claude Code's own @-import semantics) into the contents of `<path>`. `generate-claude-md(policy_source, header_source)` composes CLAUDE.md content from a policy header JSON file plus @-imports of each policy file under `policy_source`; it is a pure-Python reimplementation independent of `rabbit-cage/scripts/generate-claude-md.py`. `call_producer` raises `KeyError` on unknown producer name. Path args are resolved relative to the repo root via a shared `_resolve` helper. The `compose-template` producer is documented as a deferred future addition. Hand-rolled, stdlib only. Coverage enforced by `test/test-producers-{dispatch,read-file,expand-at-imports,generate-claude-md,publish-generated-integration}.py`. 60. **Templates validate against their sibling schemas.** Every `<X>-template.json` file under `.claude/features/contract/templates/` whose name has a corresponding `<X>.schema.json` (or `<X>.json.schema.json`) sibling under `.claude/features/contract/schemas/` MUST be a structurally-valid instance of that schema. Concretely: `templates/project-map-template.json` MUST validate against `schemas/project-map.json.schema.json`, and `templates/feature-json-template.json` MUST validate against `schemas/feature.json.schema.json` (the latter already enforced by Inv 22; this invariant promotes the pattern to a cross-template principle). Templates carry placeholder values for required fields and remain shape-valid even with placeholder strings substituted in. The shape is enforced by `test/test-templates-validate-against-schemas.py` using hand-rolled stdlib validation matching the existing schema-shape test pattern (no `jsonschema` dependency).

61. **Deployed SKILL.md copies match source.** Every `SKILL.md` file under `.claude/features/*/skills/*/SKILL.md` MUST have a byte-identical deployed copy at `.claude/skills/<skill-name>/SKILL.md` (matched by skill directory name). This invariant catches future source-only edits to a skill that fail to redeploy, preventing the deployed copy from drifting away from its declared source-of-truth. The check lives in the contract feature per the existing cross-feature lint pattern (cf. Inv 49). The shape is enforced by `test/test-deployed-skills-match-source.py` which walks every source `SKILL.md` and asserts a matching deployed copy exists with identical bytes; the test FAILs on any missing deployment or byte-difference.

## Template marker convention

Every template file in `templates/` MUST carry exactly one `template_version` marker. The marker placement depends on the template's body language:

- **JSON templates** — top-level `"template_version"` field (e.g. `bug-template.json`, `feature-json-template.json`).
- **Markdown templates with YAML frontmatter** — `template_version:` key inside the frontmatter (e.g. `skill-template.md`, `command-template.md`).
- **Markdown templates without frontmatter** — HTML comment `<!-- template_version: X.Y.Z -->` on the first non-empty line (e.g. `handoff-template.md`, `triage-template.md`, `spec-update-template.txt`).
- **Markdown templates with Jinja-style placeholders** — `{# template_version: X.Y.Z #}` on the first line (e.g. `spec-template.md`, `contract-template.md`).
- **Plain-text templates** — `# template_version: X.Y.Z` comment on a leading line (e.g. `spec-update-template.txt` and the per-callable templates under `templates/prompts/`).

`handoff-template.md` is the reference marker for HTML-comment style.

## CLI Naming Convention

All CLI flags, subcommand values, and configuration variable names in the
rabbit workflow follow two rules. These rules apply to scripts under any
feature's `scripts/`, `skills/<name>/scripts/`, and `hooks/`, and to
subcommand values exposed by skills like `/rabbit-config`.

**Rule 1 — Boolean values use `true`/`false` exclusively.**

Boolean CLI flag values and subcommand values use the literal strings
`true` and `false`. Do NOT use:

- `enabled` / `disabled`
- `on` / `off`
- `yes` / `no`
- `lock` / `unlock` for boolean state (action verbs are fine when the
  command itself is the action — e.g. `permissions lock` is an action,
  not a boolean value)

Examples:
- PREFER: `/rabbit-config human-approval true`
- AVOID:  `/rabbit-config human-approval enabled`

**Rule 2 — Names must be positive-streamlined.**

Flag names, subcommand names, and configuration variable names describe
what is present or enabled, not what is absent or disabled. If a name
would otherwise begin with `no-`, `disable-`, `skip-`, `without-`, or
any negating prefix, rewrite it to describe what IS active when the
flag/variable is true.

Examples:
- PREFER: `--code-review-full-loop`, `--enable-review`, `bypass-marker-present`
- AVOID:  `--no-code-review`, `--disable-review`, `--skip-review`

The rule is mechanical: read the name without context. If the name
describes an absence or negation, rename it. Boolean state is encoded
in the value (`true`/`false`), never in the name.

**Rationale.** Negated names compose poorly with boolean values
(`--no-code-review false` means "do require code review" — two
negations to parse). Positive names compose cleanly
(`/rabbit-config human-approval false` means "no gate"). Consistent
`true`/`false` values mean no per-flag value vocabulary to memorize.
