## Invariants

Numbering is strictly increasing and CONTIGUOUS (1..N, no holes): contract opts into the strict contiguous tier via `feature.json "contiguous_invariants": true`, so `check_invariant_monotonic_order` rejects any gap here. When an invariant is removed, the survivors are renumbered to close the hole by the deterministic reflow tool `scripts/reflow-invariants.py`, which atomically rewrites every live `Inv N` cross-reference across this feature's surfaces; the numbers of removed invariants and their history live in `CHANGELOG.md`. The contiguous tier is per-feature opt-in (mirroring the Inv 41 strict housekeeping tier): features that have not declared `contiguous_invariants` keep the baseline strictly-increasing rule (gaps tolerated) and migrate one at a time by reflowing and setting the flag.

20. `find-feature.py` MUST close all opened file handles (use `with open()` context managers) and MUST scan only canonical feature-root locations — the "no masquerading" guarantee is preserved by enumerating only documented paths, never any directory whose basename happens to be `features`. Plugin-mode detection accepts the `--repo` argument as EITHER the host-project root OR the rabbit install root (`RABBIT_ROOT` — the `.rabbit/` install dir), and resolves the canonical `rabbit_root` from whichever was supplied:

    (a) **Detect mode + resolve rabbit_root.** Plugin mode is detected by checking two candidate `mode` markers in a FIXED precedence order, and validating the resolved `rabbit_root` BEFORE returning it. The order and validation matter — they guard against the rogue-`.rabbit/.rabbit/` failure mode described below.
        - **First** check `<repo>/.runtime/mode` (the canonical RABBIT_ROOT-as-`repo` case per Inv 39; the most common caller pattern). If it exists with content `"plugin"`, candidate `rabbit_root = <repo>`.
        - **Then** check `<repo>/.rabbit/.runtime/mode` (the host-as-`repo` case). If it exists with content `"plugin"`, candidate `rabbit_root = <repo>/.rabbit`.
        - **Validate before accept.** A candidate `rabbit_root` is accepted ONLY if `<rabbit_root>/.claude/` exists as a directory — this rejects rogue `.rabbit/` subdirectories accidentally created inside the install (e.g. when a skill writes to a relative `.rabbit/...` path while CWD is already `<rabbit_root>`, creating `<rabbit_root>/.rabbit/.runtime/...` that looks like a valid plugin marker but points at an empty bogus root). If validation fails, fall through to the next candidate.
        - Otherwise standalone mode; `rabbit_root` is unused.
        The precedence (`.runtime/mode` first, `.rabbit/.runtime/mode` second) + validation closes the greedy first-match failure mode. Reversing to check `.rabbit/.runtime/mode` first would silently prefer a bogus inner `.rabbit/` over the real outer one. With the new order, the only way to mis-detect is for BOTH the outer and rogue inner roots to have valid `.claude/` directories — which is exponentially less likely than the rogue-empty-`.rabbit/` case observed in practice.

    (b) **Standalone-mode scan** (no plugin marker matched). Scan ONLY `<repo>/.claude/features/<name>/feature.json` (alphabetical). Behavior is byte-identical to the pre-amendment standalone form and remains pinned by the standalone-mode tests.

    (c) **Plugin-mode scan** (either detection path matched). Scan TWO canonical locations, yielded in this order, alphabetical within each:
        - `<rabbit_root>/.claude/features/<name>/feature.json` (rabbit-internal MVP features such as rabbit-cage, rabbit-feature, contract).
        - `<rabbit_root>/rabbit-project/features/<name>/feature.json` (per-project plugin features scaffolded under `<host>/.rabbit/rabbit-project/features/`).
        The host-project `<host>/.claude/features/` location is NEVER scanned in plugin mode (it does not exist in a plugin install; user-project sources live elsewhere). Duplicate feature names across the two plugin trees are surfaced as-is; `find-feature.py` does NOT deduplicate, so callers that need uniqueness must enforce it.

    Dual-call convention: callers (e.g. `resolve-scope.py`, `dispatch-tdd-subagent.py`) MAY pass either the host-root or the rabbit-root as `--repo` and receive the same result set in plugin mode. This removes the ambiguity in which the caller had to pre-compute the "right" repo arg per mode, while the locator script (which lives inside `.rabbit/.claude/features/contract/scripts/`) and the feature-scan target (which lives under `.rabbit/rabbit-project/features/`) required different roots. With dual-detection, either choice works.

    Enforced by `test/test-find-feature-plugin-mode.py` (extended) covering: (i) standalone unchanged; (ii) plugin via host-root (existing); (iii) plugin via rabbit-root (mode marker at `<repo>/.runtime/mode`); (iv) both rabbit-internal MVP features AND user-project features returned in plugin mode regardless of which root was passed; (v) **rogue-inner-root regression** — when called with `repo=<rabbit_root>` AND a rogue `<rabbit_root>/.rabbit/.runtime/mode` marker with content `plugin` exists, the function MUST prefer the outer `<repo>/.runtime/mode` (since it's checked first) AND the validation step (`<rabbit_root>/.claude/` directory must exist) MUST reject the bogus inner candidate — assert the returned `rabbit_root` is `<rabbit_root>`, NOT `<rabbit_root>/.rabbit`.

30. **Cross-feature invariant-numbering enforcement (two-tier, with per-feature contiguous opt-in).** Contract MUST expose `check_invariant_monotonic_order(feature_dirs: list[str]) -> CheckResult` in `lib/checks.py`, a CLI shim at `.claude/features/contract/scripts/enforcement/check-invariant-monotonic-order.py`, and a regression test `test/test-check-invariant-monotonic-order.py` wired into `test/run.py`. The check parses each feature's `spec.md` (resolved via the dual-read resolver — `docs/spec.md` preferred, `specs/spec.md` fallback), walks its `## Invariants` / `### Invariants` sections, extracts top-level numbered items (`^(\d+)\.`), and enforces TWO tiers:

    - **Baseline tier (all features).** Numbers MUST appear in strictly increasing order WITHIN each section; only repeats and decreases fail. Gaps are tolerated at this tier (they record invariant numbers removed and documented in `CHANGELOG.md`).
    - **Strict contiguous tier (opted-in features only).** A feature that declares `"contiguous_invariants": true` at the top level of its OWN `feature.json` is ADDITIONALLY required to number its invariants contiguously 1..N — a hole is a FAILURE. The check reads each feature's `feature.json` and enforces the contiguous tier on a feature iff that flag is `true` (absent/false ⇒ baseline tier only). Because the flag lives in the feature's own `feature.json`, opting in is a SINGLE-FEATURE, self-verifying touch (mirroring the Inv 41 `housekeeping_clean` strict-tier opt-in): a feature reflows its numbering and sets the flag in the same scope, so introducing the contiguous tier is NON-BREAKING (it enforces on nothing until a feature opts in) and reaches full repo-wide contiguity as each feature migrates. The test MUST be self-testable: it accepts a `RABBIT_CONTIGUOUS_INVARIANTS` env override (comma-separated feature names) that REPLACES the `feature.json`-derived opt-in set so a hermetic fixture test can drive the contiguous tier without writing `feature.json`; absent the override the opt-in is read from each feature's `feature.json contiguous_invariants` flag.

    A `KNOWN_ISSUES` allowlist inside the library function permits features pending a renumber to be skipped from enforcement; the current value is `[]`. Features absent from `KNOWN_ISSUES` are validated; any violation FAILs the test with a clear `<feature>:<section>: <prev_num> → <next_num>` diagnostic (the contiguous-tier diagnostic additionally names the missing number(s) and points at `scripts/reflow-invariants.py`). Features added later are validated automatically. **Reflow tooling.** Because the contiguous tier means removing an invariant forces renumbering every higher invariant AND rewriting every `Inv N` cross-reference, contract MUST provide a DETERMINISTIC reflow tool — `lib/reflow.py` (`reflow_feature(feature_dir, *, dry_run=False) -> ReflowResult`) plus the CLI shim `scripts/reflow-invariants.py` — that renumbers a single feature's invariants to contiguous 1..N and atomically rewrites every LIVE `Inv N` cross-reference (in a single pass, no cascade) across that feature's own `spec.md`, `contract.md`, `skills/*/SKILL.md`, `lib/`, `scripts/`, `test/`, and `templates/` surfaces, NEVER touching `CHANGELOG.md` (the point-in-time numbering record) and NEVER remapping a removed (gap) invariant number. It is enforced by `test/test-reflow-invariants.py`; the contiguous tier and the opt-in are enforced by `test/test-invariant-contiguous-opt-in.py`.

The range's LOW bound is the next scheduled fire minute at/after `now`; the HIGH bound is that fire plus the bounded CronCreate scheduler jitter the runtime adds to recurring fires (up to 10% of the cadence period, capped at 15 minutes, floored at 1 minute). A bare single-point ETA reads systematically EARLY because it ignores that jitter plus idle-check latency, so the range form makes the estimate honest and never early. This range shape MUST match the rabbit-auto-evolve `banner-status.py` SessionStart wording so the banner and the Stop line read consistently. When the cadence source is absent, unreadable, or yields no parseable cron, the idle line degrades gracefully to the bare base text `"auto-evolve loop active — idle between ticks"` (no crash, no fabricated ETA). The ETA extension applies ONLY to this idle line; the four priority-marker lines, the restart-pending line, and the active-absent `[]` case are unchanged.

    **Next-tick ETA computation.** The ETA is derived from the heartbeat cadence — the durable CronCreate heartbeat that drives `/rabbit-auto-evolve tick`. The helper:
    - reads `<repo_root>/.claude/scheduled_tasks.json`; on missing file, unreadable file, JSON parse error, no matching task, or no parseable `cron` → returns `None` (caller renders the bare idle line).
    - parses the `cron` minute and hour fields enough to find the next wall-clock minute at or after `now` (an INJECTED `datetime` for determinism — the helper accepts `now` rather than calling the clock itself, so tests pin a fixed `now`; `emit_auto_evolve_stop_line` passes its own injected `now`, defaulting to `datetime.datetime.now()` only when none is given). Supports the cadence forms actually used by the heartbeat: minute lists/steps (`13,43`, `*/15`, `*`) with hour `*`.
    - derives the cadence PERIOD from the parsed fire-minute set (the smallest gap between consecutive fire minutes, wrapping across the hour; a single fire minute means once-per-hour, period 60) and computes the bounded scheduler jitter as `max(1, min(15, ceil(period * 0.10)))` minutes. Renders the jitter-inclusive RANGE string `"~HH:MM–HH:MM (scheduler jitter)"`, where the LOW bound is the next fire and the HIGH bound is that fire plus the jitter (the upper bound may wrap across the hour or midnight). This matches the rabbit-auto-evolve `banner-status.py` range form.

## Template marker convention

Every template file in `templates/` MUST carry exactly one `template_version` marker. The marker placement depends on the template's body language:

- **JSON templates** — top-level `"template_version"` field (e.g. `bug-template.json`, `feature-json-template.json`).
- **Markdown templates with YAML frontmatter** — `template_version:` key inside the frontmatter (e.g. `skill-template.md`, `command-template.md`).
- **Markdown templates without frontmatter** — HTML comment `<!-- template_version: X.Y.Z -->` on the first non-empty line (e.g. `handoff-template.md`, `triage-template.md`, `spec-update-template.txt`).
- **Markdown templates with Jinja-style placeholders** — `{# template_version: X.Y.Z #}` on the first line (e.g. `spec-template.md`, `contract-template.md`).
- **Plain-text templates** — `# template_version: X.Y.Z` comment on a leading line (e.g. `spec-update-template.txt` and the per-callable templates under `templates/prompts/`).

`handoff-template.md` is the reference marker for HTML-comment style.

## CLI Naming Convention

All CLI flags, subcommand values, and configuration variable names in the
rabbit workflow follow two rules. These rules apply to scripts under any
feature's `scripts/`, `skills/<name>/scripts/`, and `hooks/`, and to
subcommand values exposed by per-feature config commands like
`/rabbit-cage-config`.

**Rule 1 — Boolean values use `true`/`false` exclusively.**

Boolean CLI flag values and subcommand values use the literal strings
`true` and `false`. Do NOT use:

- `enabled` / `disabled`
- `on` / `off`
- `yes` / `no`
- `lock` / `unlock` for boolean state (action verbs are fine when the
  command itself is the action — e.g. `permissions lock` is an action,
  not a boolean value)

Examples:
- PREFER: `/rabbit-cage-config bypass-permissions true`
- AVOID:  `/rabbit-cage-config bypass-permissions enabled`

**Rule 2 — Names must be positive-streamlined.**

Flag names, subcommand names, and configuration variable names describe
what is present or enabled, not what is absent or disabled. If a name
would otherwise begin with `no-`, `disable-`, `skip-`, `without-`, or
any negating prefix, rewrite it to describe what IS active when the
flag/variable is true.

Examples:
- PREFER: `--code-review-full-loop`, `--enable-review`, `bypass-marker-present`
- AVOID:  `--no-code-review`, `--disable-review`, `--skip-review`

The rule is mechanical: read the name without context. If the name
describes an absence or negation, rename it. Boolean state is encoded
in the value (`true`/`false`), never in the name.

**Rationale.** Negated names compose poorly with boolean values
(`--no-code-review false` means "do require code review" — two
negations to parse). Positive names compose cleanly
