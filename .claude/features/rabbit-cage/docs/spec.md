# rabbit-cage — Spec

## Purpose

rabbit-cage is the service dispatcher of the rabbit workflow. It owns four
Claude Code event hooks and the bootstrap installer. Three of the hooks are
generic per-event dispatchers that enumerate every feature's
`feature.json runtime[<event>]` declarations and invoke each declared API via
`contract.lib.runtime`; the fourth (PreToolUse) is `scope-guard.py`, owned
outright by rabbit-cage. The installer enumerates every feature's
`feature.json manifest` and invokes each call via `contract.lib.publish`.

rabbit-cage holds no per-feature integration code. Its own
MANIFEST / RUNTIME / CONFIGURATION declarations live in
`.claude/features/rabbit-cage/feature.json` and are consumed by the same
dispatchers and installer that consume every other feature's declarations.

## Surface

The deployed surface (produced by `install.py` running rabbit-cage's MANIFEST):

- `.claude/hooks/scope-guard.py` — PreToolUse
- `.claude/hooks/stop-dispatcher.py` — Stop
- `.claude/hooks/session-start-dispatcher.py` — SessionStart
- `.claude/hooks/user-prompt-submit-dispatcher.py` — UserPromptSubmit
- `.claude/hooks/_dispatcher_lib.py` — shared helper imported by the three
  event-dispatcher hooks above
- `.claude/settings.json` — Claude Code config (env, permissions, hooks)
- `.claude/commands/rabbit-refresh.md`, `.claude/commands/rabbit-project.md`,
  `.claude/commands/rabbit-update.md`
- `CLAUDE.md` — generated by the `generate-claude-md` content producer
- `README.md`, `install.py`, `install.sh` — committed copies at the repo root; `install.sh` is the one-liner shell wrapper that drives `install.py main()` after extracting the upstream tarball

## Dispatcher Behavior

Each event dispatcher (`stop-dispatcher.py`, `session-start-dispatcher.py`,
`user-prompt-submit-dispatcher.py`):

1. Enumerates every feature under `.claude/features/` in alphabetical order
   by directory name, skipping features whose `feature.json status` the
   contract status enum marks end-of-life (contract Inv 34) and feature
   directories whose `feature.json` is missing or malformed.
2. For each active feature, iterates `feature.json runtime[<event>]` in
   declaration order. For each entry `{api, args}`, invokes
   `contract.lib.runtime.<api>(**args, repo_root=<root>)`, forwarding
   `feature_dir=<fdir>` only when the runtime API's signature declares it.
3. Partitions the typed results: `banner` results are rendered via
   `rabbit_print.rabbit_print(text, icon, color, format="banner")`
   (decorated with ━━━ bars; the dispatcher reads `text`, `icon`, and
   `color` directly from the payload); `print` results are rendered via
   `rabbit_print.rabbit_subline(text, color, icon)` (compact format);
   `subline` results are rendered via
   `rabbit_print.rabbit_subline(text, color)` without icon. Rendered
   `banner`/`print`/`subline` lines are ordered by a stable footer
   partition (Inv 31): lines whose payload carries `order == "footer"`
   are held back and appended AFTER all non-footer lines, each group
   preserving dispatch order. All rendered lines are then joined into
   `systemMessage` via `rabbit_print.rabbit_block`; `inject` results are
   concatenated into `additionalContext` (inject ordering is unaffected
   by the footer partition); `ok` results are dropped; `error` results
   are written to stderr (one line each) and never surfaced.
4. Emits at most one JSON object to stdout per invocation. When no `print`
   and no `inject` result is collected, the dispatcher writes nothing
   (exit 0).

A runtime API named in a declaration but not exported by
`contract.lib.runtime` produces an `error` result (logged to stderr); the
dispatcher continues to the next entry.

## Installer Behavior

`install.py` is the user-facing MVP installer, driven by `install.sh` (the one-liner shell wrapper). Usage: `install.py --src <extracted-tarball-dir> --target <project>/.rabbit`. It lays down the minimum file closure needed for the user-promised surfaces — drift-protected Claude on session start, `rabbit-feature-scaffold <name> <path-glob>` for opt-in feature mapping, scope-guard plugin-mode blocking, and the `.rabbit/.runtime/scope-bypass-once` override marker.

`main()` copies from explicit `(source, dest)` tuples declared at module top — `SAME_PATH_FILES`, `HOOKS`, `SKILLS`, `AGENTS`, `COMMANDS`, `FEATURE_INCLUDES`. After the closure copy and the Inv 19 settings rewrite, `main()` canonicalizes the installed surfaces by running the publish flow against the install (Inv 43), so the installed manifest surfaces are byte-identical to what `check_manifest_drift` republishes at runtime. The MVP closure excludes development surfaces: `test/`, `docs/`, `scripts/enforcement/`, end-of-life tombstone features the contract status enum marks (contract Inv 34; e.g. tdd-state-machine), and prompt templates corresponding to skills/subagents NOT shipped via `SKILLS`/`AGENTS` (templates for shipped skills MUST be included in `FEATURE_INCLUDES["contract"]` per Inv 23). The installer refuses if `--target` exists and is non-empty UNLESS `--update` is passed (see Inv 22 for update-mode semantics), and exits non-zero on any missing required source file. The `COMMANDS` list MUST include every command that rabbit-cage's manifest deploys (`rabbit-refresh.md`, `rabbit-project.md`, `rabbit-update.md`); the `SKILLS` list MUST include every skill that rabbit-feature's manifest deploys (the four `rabbit-feature-*` skills: `touch`, `scope`, `new`, `audit`), every skill that rabbit-spec's manifest deploys (`rabbit-spec-update`); rabbit-spec's `scripts/dispatch-spec-creator.py` input assembler and its `rabbit-spec-creator` agent are deployed via `FEATURE_INCLUDES["rabbit-spec"]` / `AGENTS`, and the `rabbit-issue` skill (so plugin users can file bugs from inside their project — the very window in which they are most likely to find them); the `AGENTS` list MUST include the `tdd-subagent` agent (so `rabbit-feature-touch` Step 5 can dispatch the TDD subagent in plugin mode — without it the entire feature-touch flow halts at Step 5); and `SAME_PATH_FILES` MUST include `README.md`. The complete initial deployment is required so that `check_manifest_drift` finds all expected artifacts already present on the very first Stop event — if any are missing, `check_manifest_drift` deploys them and reports spurious drift. Completeness is verified by the e2e test `test-feature-includes-manifest-closure.py` (per Inv 21).

After copying the file closure, `main()` rewrites `<target>/.claude/settings.json` per Inv 19 (sets `env.RABBIT_ROOT` to the absolute target path; replaces every `$(git rev-parse --show-toplevel)` inside any `hooks[<event>][].hooks[].command` string with `$RABBIT_ROOT`), then writes two generated files: `<target>/.gitignore` (rabbit-owned ephemerals — `.runtime/`, `prompts/`, `tdd-report-*.json`, `impl-suggestion-*.json`, scope markers, `__pycache__/`, `*.pyc`) and `<target>/.version` (install pin label from `RABBIT_INSTALLED_REF` env var; when that env var is unset or empty — the local `--src` install case — `main()` derives a meaningful local marker per Inv 48 instead of a sentinel). On a successful `--update`, after the install completes `main()` emits a brief deterministic post-update changelog summary naming the old→new version range and the intervening live-track `vX.Y.Z` git-tag entries from the source tree (Inv 46).

`run_publish_loop(target_root: str) -> int` is the publish-flow driver. It enumerates every `<target_root>/.claude/features/*/feature.json` in alphabetical order, skips features the contract status enum marks end-of-life (contract Inv 34) and features without a `manifest`, and for each manifest entry `{api, args}` invokes `contract.lib.publish.<api>(**args, feature_dir=<fdir>, repo_root=target_root)`. Continues past per-call failures and returns the failure count. `main()` invokes it (via `canonicalize_installed_surfaces`, Inv 43) to canonicalize the installed surfaces; it is ALSO imported directly by `test-deployed-hooks-execute.py`, `test-install-publish-loop.py`, and the cross-feature `test-manifest-deploys-correctly.py` tests in rabbit-feature and tdd-subagent — all import it as `from install_under_test import run_publish_loop`.

## scope-guard Semantics

`scope-guard.py` is the sole PreToolUse hook. It receives one tool
invocation payload on stdin and either allows (exit 0) or denies (exit 2)
the write.

A write is ALLOWED when any of the following holds:

- Target is outside the repo root.
- Target basename is on the filename allowlist:
  `settings.local.json`, `.gitignore`, `.rabbit-scope-override`.
- Target is the scope marker file itself
  (`.rabbit-scope-active` or `.rabbit-scope-active-*`).
- Target path is under an allowlisted prefix: `.claude/bugs/`,
  `.claude/backlogs/`, or `.rabbit/`. For each prefix, both the exact
  directory path (e.g. `.rabbit`) and any path inside it
  (e.g. `.rabbit/x.json`) match; the exact form uses strict equality so
  sibling paths such as `.rabbit-scope-active` do not match.
- Target matches the feature spec-artifact path pattern (the dual-read
  carve-out specified authoritatively by Inv 33: `spec.md` under `specs/`
  or legacy `docs/spec/`, and `spec.md`/`contract.md`/`CHANGELOG.md` under
  the flat `docs/` home).
- A per-feature scope marker `.rabbit-scope-active-<feature>` exists at the
  repo root and the target is inside that feature's directory.
- The active per-feature scope marker is `.rabbit-scope-active-rabbit-cage`
  AND the target is one of rabbit-cage's owned repo-root bootstrap files
  (`install.sh`, `install.py`, root `README.md`). rabbit-cage owns these
  files but they live OUTSIDE its feature directory (Inv 41), so this clause
  authorizes a rabbit-cage-scoped edit to them without an override. ONLY
  rabbit-cage's marker authorizes this set; another feature's marker does
  not.
- A global scope marker `.rabbit-scope-active` exists at the repo root, its
  contents name a resolvable feature, and the target is inside that
  feature's directory. When the named feature is in `tdd_state: test-green`,
  the write is DENIED unless an active `.rabbit-scope-override` is consumed.
- A `.rabbit-scope-override` file at the repo root contains `session` (ALLOW
  any write and leave the marker in place), `one-time` (ALLOW any single
  write, delete the marker, create `.rabbit-scope-override-used`), or the
  file-scoped form `one-time:<repo-relative-path>` (ALLOW a single write ONLY
  to the declared path, then delete the marker and create
  `.rabbit-scope-override-used`; a write to any OTHER path is DENIED while
  this override is present). The file-scoped form is PREFERRED over bare
  `one-time` when the target path is known (least privilege); bare `one-time`
  is RETAINED for backward compatibility (Inv 39).
- A decompose-context marker `.rabbit/.runtime/decompose-active` is present,
  un-expired, and authorizes the target's feature directory (Inv 47). The
  marker is a bounded, auto-cleared pass-through for batch work that spans
  several feature directories: while it is present, a write whose target is
  inside any feature named in the marker's authorized set is ALLOWED with no
  per-feature scope marker and no override. The marker is honored ONLY while
  present (the orchestration that set it clears it on completion) and ONLY
  until any declared expiry passes; outside the authorized set the normal
  bounded-scope rules apply, unchanged.

A scope marker that names a feature `find-feature.py` cannot resolve
DENIES the write with a message naming the unresolvable feature.

When `scope-guard.py` reaches the default-deny path (no marker, no
override, no allowlist match), the stderr message presents three explicit
options: SESSION OVERRIDE, ONE-TIME OVERRIDE, and USE rabbit-feature-touch.
Both override options require explicit in-conversation user confirmation
before any marker is written; `scope-guard.py` itself never creates
`.rabbit-scope-override`.

`scope-guard.py` is quote-aware: `extract_bash_targets()` strips heredoc
bodies (`<<` and `<<-` forms), single-quoted regions, and double-quoted
regions (the double-quote pass uses `re.DOTALL`) from the full command
string BEFORE splitting on `;|&` segment delimiters.

## Invariants

1. The four hooks deployed under `.claude/hooks/` are exactly
   `scope-guard.py`, `stop-dispatcher.py`, `session-start-dispatcher.py`,
   and `user-prompt-submit-dispatcher.py`. They are registered in
   `.claude/settings.json` under the events `PreToolUse`, `Stop`,
   `SessionStart`, and `UserPromptSubmit` respectively.
2. Each event dispatcher enumerates features alphabetically, skips
   features the contract status enum marks end-of-life (contract Inv 34),
   invokes each `runtime[<event>]` entry in declaration order via
   `contract.lib.runtime`, and emits at most one JSON object per
   invocation. Unknown APIs and per-call exceptions produce `error` results
   logged to stderr; the dispatcher does not abort the event.
3. `scope-guard.py` is the sole PreToolUse hook and the only rabbit-cage
   hook that is NOT a generic dispatcher.
4. `install.py` exports three callables. (a) `main()` is the user-facing MVP installer: parses `--src` + `--target` (both optional under `--update` — see Inv 22g for defaults) plus the optional `--update` flag, copies the MVP file closure (top-level files + deployed hooks/skills/agents + per-feature stripped sub-paths) from explicit `(source, dest)` tuples declared at module top, performs the plugin-mode `settings.json` rewrite specified in Inv 19, writes generated `.gitignore` + `.version`, canonicalizes the installed surfaces via the publish flow (Inv 43), and exits non-zero on any missing required source file. Without `--update`, refuses when `--target` exists and is non-empty (default-deny on accidental re-run, protecting committed `.rabbit/` trees from silent regression); with `--update`, the empty-target check is skipped and the in-place refresh defined in Inv 22 governs. (b) `run_publish_loop(target_root: str) -> int` is the publish-flow driver: enumerates every feature's `manifest` entries in declaration order, invokes each via `contract.lib.publish`, continues past failures, and returns the failure count. (c) `canonicalize_installed_surfaces(dst_root)` wraps `run_publish_loop` with graceful degradation (Inv 43) and is the call `main()` makes; the dev-test suites import `run_publish_loop` directly as `from install_under_test import run_publish_loop`. All three callables are stdlib-only (argparse, json, os, shutil, sys, pathlib).
5. `scope-guard.py` evaluates writes per the rules in "scope-guard
   Semantics" above; a scope marker that names an unresolvable feature
   DENIES.
6. `scope-guard.py` `extract_bash_targets()` strips heredoc bodies and
   quoted regions from the full command string before splitting on `;|&`.
7. `scope-guard-on.py` at `scripts/scope-guard-on.py` deletes
   `.rabbit-scope-override` (if present) and prints a confirmation; it is a
   no-op when no override marker is present. It is wrapped by a `scope-guard`
   configurable in `feature.json configuration` (subcommand `scope-guard`,
   value `on` → `delete_marker` on `.rabbit-scope-override`) so
   `/rabbit-cage-config scope-guard on` revokes a session override via the
   data-driven config interpreter. That configurable carries no
   `alert-on`/`alert-message` (no double-alert): the `.rabbit-scope-override`
   `check_marker_alert` entries (Inv 16) and the default-deny SESSION OVERRIDE
   clause both inline the `/rabbit-cage-config scope-guard on` revoke hint. Every
   user-facing revoke instruction surfaces that command, NOT the raw script
   path.
8. `feature.json` declares `manifest`, `runtime` (with the three keys
   `Stop`, `SessionStart`, `UserPromptSubmit`), and `configuration` arrays
   as described in this spec. Every API name listed in any of those
   declarations is exported by the corresponding `contract.lib.*` module.
   The live `feature.json` validates against the meta-contract schemas
   under `contract/schemas/` (verified by invoking
   `contract/scripts/validate-meta-contract.py` against the feature
   directory).
9. Every runtime marker written under the repo root MUST be listed in
   `.gitignore`. The mandated set: `.rabbit-prompt-counter`,
   `.rabbit-sync-counter`, `.rabbit-scope-active`, `.rabbit-scope-active-*`,
   `.rabbit-scope-override`, `.rabbit-scope-override-used`,
   `.rabbit-skills-updated`, `.rabbit-human-approval-bypass`.
10. Every script under `hooks/` and `scripts/` is a standalone executable
    Python 3 file (`#!/usr/bin/env python3`) using stdlib only. No `.sh`
    file exists in the rabbit-cage feature directory. Tests under `test/`
    are Python (`.py`).
11. The rabbit-cage feature version MUST be identical across `feature.json
    version`, `specs/spec.md` frontmatter `version`, and
    `specs/contract.md` frontmatter `version`. Script-level `Version:`
    docstring markers under `hooks/` and `scripts/` are NOT bound by this
    invariant: they track the script artifact's own SemVer and evolve
    independently of feature.json. The three-way feature-version pin
    governs the public surface (spec + contract + manifest); per-script
    versions govern internal sub-component evolution.
12. Every helper module that an event-dispatcher hook imports at runtime
    (currently `_dispatcher_lib.py`) MUST be deployed alongside the
    dispatchers under `.claude/hooks/` via a rabbit-cage `MANIFEST` entry.
    The dispatchers resolve helpers from their own directory; a helper
    that exists only at the source path will fail to import at the
    deployed location.
13. Every `icon` value passed to `contract.lib.runtime` APIs (alert dicts
    in RUNTIME entries) and every `alert-message.icon` value in
    CONFIGURATION entries MUST be a literal renderable Unicode glyph
    (e.g. `🔑`, `🚨`, `🔓`, `🔄`, `✨`, `⚠️`), NOT a lookup name
    (`"key"`, `"siren"`, `"unlock"`, …). `rabbit_subline(text, color, icon)`
    and `rabbit_print(text, icon, color, format)` interpolate `icon`
    literally; neither performs name-to-glyph resolution. (Per contract
    Inv 48 there is no message-id registry; every producer
    supplies `text`, `icon`, and `color` inline at the call site.)
14. rabbit-cage owns the cross-cutting CLAUDE.md `@`-import resolution
    tests. The two assertions — (a) every `@`-import in any `CLAUDE.md`
    resolves to an existing file, and (b) no `CLAUDE.md` carries a
    stale `@`-import to a removed source — live under
    `.claude/features/rabbit-cage/test/` because rabbit-cage owns the
    `generate-claude-md` content producer that emits `CLAUDE.md`. The
    policy feature owns the rule files those imports point at, but not
    the import-resolution invariant itself.
15. Every hook command string registered in `.claude/settings.json` (the
    `hooks[<event>][].hooks[].command` field) MUST be CWD-independent.
    The source `settings.json` shipped from this feature begins each
    command with `$(git rev-parse --show-toplevel)/` so that `/bin/sh`
    resolves the absolute path to `.claude/hooks/<name>.py` at hook-fire
    time regardless of the session's current working directory in the
    standalone workspace (where a `.git` directory exists at the repo
    root). A bare relative path such as `.claude/hooks/scope-guard.py`
    is forbidden because Claude Code's Bash tool persists CWD between
    calls; after any `cd` into a subdirectory the relative path resolves
    outside the repo and the hook silently fails (non-blocking
    `No such file or directory`). Emission of the command string is the
    responsibility of `contract.lib.publish.publish_hook`; rabbit-cage
    owns the user-visible requirement that the deployed registration
    fire correctly from any CWD. In plugin installs the `git rev-parse`
    form is unsuitable (the install target is typically not itself a git
    repo, and even when its parent is, the resolved root points outside
    `.rabbit/`); Inv 19 governs the plugin-mode rewrite.
16. rabbit-cage's `feature.json runtime.SessionStart` declares SIX entries
    in order: (1) `welcome_with_policy` (existing), (2) `write_mode_marker`
    (args `{}`), (3) `check_release_update` (args `{}`), (4)
    `check_marker_alert` for the STANDALONE session-override marker (args
    `{path: '.rabbit-scope-override', content: 'session',
    alert: {text: 'SCOPE GUARD OFF (session override active)',
    icon: '🔓', color: 'red'}}`), (5) `check_marker_alert` for the PLUGIN
    session-override marker (args `{path: '.rabbit/.rabbit-scope-override',
    content: 'session', alert: {…same banner…}}`) — entries (4) and (5)
    together surface the active session-override alert at session-start in
    BOTH modes (Inv 25 makes the marker's canonical location per-mode:
    `<rabbit_root>/.rabbit/.rabbit-scope-override` in plugin mode,
    `<repo_root>/.rabbit-scope-override` in standalone; a single relative
    `check_marker_alert` path resolves to only ONE of them against
    `repo_root`, so both paths MUST be declared for the banner to fire in
    either mode), in addition to the symmetric Stop-event alert —
    and (6) `emit_configurable_alert`
    (args `{feature_name: 'rabbit-cage', configurable_id:
    'bypass-permissions'}`), the per-feature bypass-permissions
    active-override alert (Inv 40c). The two `check_marker_alert` entries
    never double-fire: the marker lives at exactly ONE canonical location
    per mode and `check_marker_alert` no-ops on an absent marker. The
    SessionStart dispatcher invokes all six in declaration order via
    `contract.lib.runtime`. The same dual-path pair is declared in
    `runtime.Stop`. The
    `write_mode_marker` API (owned by
    `contract.lib.runtime` and built on top of `rabbit-meta.lib.mode_detection.detect_mode`)
    detects whether rabbit is running in `"plugin"` or `"standalone"` mode
    and writes the result to `<repo_root>/.rabbit/.runtime/mode` for
    downstream consumers. The `check_release_update` API (owned by
    `contract.lib.runtime` per contract Inv 47 / Inv 63) checks the upstream
    release channel against the local `.version` file (8h-throttled, network
    failure silent), and emits an update-available `print_result` when a
    newer ref is published on the same channel — letting plugin users
    discover new releases at SessionStart without polling out-of-band.
    rabbit-cage owns ONLY the wiring (the `runtime.SessionStart` declaration
    that fires the APIs); detection logic for `write_mode_marker` is owned
    by rabbit-meta, the helper script + fetch logic for `check_release_update`
    is owned by contract, and the API implementations are both owned by
    contract. The `welcome_with_policy` entry's `sublines` array carries
    EXACTLY the three policy-summary sublines (philosophy.md / spec-rules.md /
    coding-rules.md) and NO permission-bypass info subline. The permission-bypass
    guidance is NOT advertised on startup — it is useful but non-urgent
    information that would otherwise print on EVERY session start (including a
    fresh `.rabbit` install). The guidance is instead surfaced ON-DEMAND through
    the `/rabbit-cage-config` config-query path (Inv 40 clause (e)), not the
    automatic SessionStart path. This is
    distinct from the scope-override SAFETY notices (Inv 16 entries (4)/(5),
    Inv 25), which are safety alerts that MUST still fire on startup when an
    override is active. Enforced by
    `test/test-bypass-permissions-on-demand-not-startup.py` (e2e via the real
    deployed dispatcher): the rendered systemMessage carries NO permission-bypass
    advert, the three policy sublines remain, the message IS emitted by the
    on-demand `rabbit-cage-config.py help` query path, and the scope-override
    SAFETY notice still fires when a `session` override is active. Wired into
    `test/run.py`.
17. **Plugin-mode scope-guard decision tree.** When `scope-guard.py` runs
    and `<repo_root>/.rabbit/.runtime/mode` exists with content `"plugin"`,
    it takes the plugin-mode branch instead of the standalone decision
    tree. The plugin-mode branch decides each target as follows:
    (a) Target inside `.rabbit/.claude/**` (with carve-outs for `.rabbit/CLAUDE.md`
    and `.rabbit/.gitignore`) → DENY always. These paths are rabbit's own
    machinery inside the plugin install and are never edited by the user-facing
    agent. Target inside `.rabbit/rabbit-project/**` is DENY always EXCEPT when
    the path matches `.rabbit/rabbit-project/features/<name>/**` (where `<name>`
    is the first path segment after `rabbit-project/features/`): such paths
    contain user-owned plugin feature artifacts (specs, contracts, feature.json
    scaffolded by `rabbit-feature-scaffold`) and MUST fall through to clauses
    (b)/(c) below — the SAME per-feature scope-marker gate that governs
    user-project source files. Presence of `<repo_root>/.rabbit/.runtime/scope-active-<name>`
    ALLOWS the write; absence DENYs with the structured three-option message
    naming the parsed feature `<name>` and the absolute target path. Writes
    to `.rabbit/rabbit-project/` paths NOT matching `features/<name>/**`
    (e.g. `.rabbit/rabbit-project/project-map.json` itself, or future
    rabbit-project metadata files) remain always-DENY.
    (a2) **Plugin spec-artifact path-pattern carve-out.** When the target path matches the spec-artifact dual-read carve-out (specified authoritatively by Inv 33) under `<rabbit_root>/rabbit-project/features/<name>/` (`<name>` is `[^/]+`): ALLOW unconditionally, regardless of scope-marker state. This enables the `rabbit-spec-creator` subagent to write initial spec bodies to freshly scaffolded plugin features without a scope marker (the `rabbit-decompose` Step 4 batch pipeline depends on it). The carve-out is NARROW: only the Inv 33 basenames under their recognized parent directories. Other writes inside the same feature dir (`feature.json`, `scripts/`, `tests/`, a `specs/contract.md`, an arbitrary `docs/<other>.md`, etc.) still flow through the per-feature scope-marker gate in clauses (b)/(c). This clause is evaluated BEFORE clauses (b)/(c) so the spec-artifact ALLOW takes precedence over the marker-required default. Enforced by `test/test-plugin-scope-guard-allows-fresh-feature-spec-md.py`, `test/test-plugin-scope-guard-denies-non-spec-write-without-marker.py`, `test/test-plugin-scope-guard-mid-tdd-still-requires-marker.py`, `test/test-standalone-spec-md-carveout-unchanged.py`, `test/test-scope-guard-specs-dir-dual-read.py`, and `test/test-scope-guard-flat-docs-dual-read.py`.
    (b) Target matches a feature-path glob declared in
    `<repo_root>/.rabbit/rabbit-project/project-map.json` AND a per-feature
    scope marker `<repo_root>/.rabbit/.runtime/scope-active-<name>` exists
    for that feature → ALLOW.
    (c) Target matches a declared feature path AND no marker exists for
    that feature → DENY with a structured three-option message: SESSION
    OVERRIDE, ONE-TIME OVERRIDE (via the `scope-bypass-once` marker, Inv
    18), and USE rabbit-feature-touch (recommended). The DENY message
    names the matched feature and the absolute target path.
    (d) Target matches no declared feature path → ALLOW (default safe: a
    user file outside any declared feature is unrestricted).
    When `.rabbit/.runtime/mode` is absent or its content is not exactly
    `"plugin"`, the standalone decision tree (Inv 5) governs unchanged.
    The library `.claude/features/rabbit-cage/lib/project_map_reader.py`
    owns map I/O and path matching, exporting two functions:
    `load_map(repo_root: str) -> dict | None` (reads

18. **One-shot bypass-once marker.** Before evaluating any decision in

19. **Plugin-install settings.json rewrite.** After copying the source
    `.claude/settings.json` to `<target>/.claude/settings.json`,
    `install.py main()` performs two in-place edits on the destination
    file:
    (a) Sets `env.RABBIT_ROOT` to the absolute path of `<target>` (the
    plugin install root, e.g. `/path/to/project/.rabbit`). The `env`
    block is created if absent; an existing `RABBIT_ROOT` key is
    overwritten.
    (b) Replaces every literal occurrence of
    `$(git rev-parse --show-toplevel)` with `$RABBIT_ROOT` inside any
    `hooks[<event>][].hooks[].command` string. No other fields are
    touched. The replacement is a literal substring substitution; the
    surrounding `/.claude/hooks/<name>.py` suffix is preserved verbatim.
    (c) Applies the same two edits to the feature-local source copy at
    `<target>/.claude/features/rabbit-cage/settings.json` (if that file
    exists in the install). This is required so that `check_manifest_drift`
    (which regenerates the deployed copy by running `publish_settings` from
    the feature-local source) produces a result identical to the deployed
    copy — making the rewrite idempotent across sessions. Without (c),
    `check_manifest_drift` reads the unmodified source (original git-based
    hook commands, no `RABBIT_ROOT`), detects drift versus the rewritten
    deployed copy, overwrites the deployed copy with the source content, and
    silently reverts the RABBIT_ROOT injection on every Stop event.
    All three edits are idempotent: re-running on already-rewritten files
    is a no-op.

22. **Install-time `--update` mode for in-place refresh.** `install.py main()` accepts an optional `--update` flag — the SOLE update mechanism. `install.sh` does NOT support `--update`; install.sh is for first-time installs only. Update-mode semantics:
    (a) **install.sh refuses unconditionally on existing target; install.py default-deny without the flag.** `install.sh` ALWAYS refuses when `.rabbit/` exists at the invocation cwd (no `--update` flag, no `RABBIT_UPDATE` env-var equivalent). The refusal error message MUST point the user at the survivor command: `python3 .rabbit/install.py --update`. `install.py` without `--update` refuses if `--target` exists and is non-empty (unchanged). Accidental re-run never silently overwrites a committed install.
    (b) **Refresh in place when set.** When `--update` is passed, the empty-target check is skipped. `install.py` refreshes every file inside the MVP install closure (the union of `SAME_PATH_FILES`, `HOOKS`, `SKILLS`, `AGENTS`, `COMMANDS`, and `FEATURE_INCLUDES`) by overwriting per-file at its declared destination — no `rm -rf` of the target tree, no whole-tree relayout.
    (c) **Preserve everything outside the closure.** Anything NOT in the install closure survives untouched. Concretely: `<target>/.rabbit/.runtime/` (counters, scope markers, bypass-once, mode marker, and any other transient bookkeeping), `<target>/.claude/settings.local.json`, and any user-touched non-rabbit files anywhere under `<target>`.
    (d) **`settings.json` via `publish_settings`.** On `--update`, the `<target>/.claude/settings.json` write routes through `contract.lib.publish.publish_settings` (contract Inv 44) instead of a raw file copy. This reuses the existing merge-aware semantics — non-rabbit `hooks[<event>][]` entries, user-added permissions, and user-set env vars survive — without duplicating the merge logic. The Inv 19 plugin-mode rewrites (RABBIT_ROOT injection, `$(git rev-parse --show-toplevel)` -> `$RABBIT_ROOT`) still apply after the merge.
    (e) **Version visibility.** Before refresh, when `<target>/.version` exists, `install.py main()` reads its content and prints `updating <old-ref> -> <new-ref>` to stdout (where `<new-ref>` is the value `write_version_pin` is about to write — `RABBIT_INSTALLED_REF` env var, default `"unknown"`). The existing `.version` file is reused; no new state file is introduced. `install.sh` MUST export `RABBIT_INSTALLED_REF` from `RABBIT_REF` (or honour an externally-set `RABBIT_INSTALLED_REF`) before invoking `install.py`, so the version label reflects the fetched ref instead of always defaulting to `"unknown"` for one-liner installs.
    (f) **Idempotency.** Two consecutive `--update` runs against the same source ref produce zero further filesystem changes on the second pass — both the in-closure overwrites (content-equality short-circuit) and the `publish_settings` merge are idempotent by construction.
    Enforced by four tests under `.claude/features/rabbit-cage/test/`: `test-install-update-mode.py` (fresh install -> mutate `.rabbit/.runtime/` and add a custom permission to settings.json -> run `install.main(--update)` -> assert runtime tree byte-identical, custom permission preserved, closure files refreshed, version pin updated), `test-install-update-idempotent.py` (two consecutive `--update` runs produce no second-pass diff), `test-install-refuses-without-update.py` (pins the default-deny refusal when `--update` absent), and `test-install-sh-update-flag.py` (bash-level smoke: `install.sh --update` and `RABBIT_UPDATE=true install.sh` both forward `--update` to `install.py`).
    (h) **Self re-exec into the freshly-copied `install.py` before the closure loop.** When `--update` is set AND `--src` was NOT explicitly supplied (i.e. the self-fetch path of Inv 22g ran), `install.py main()` MUST `os.execv` into the freshly-copied `<target>/install.py` after that file has been written to disk by the SAME_PATH_FILES loop BUT BEFORE the FEATURE_INCLUDES / HOOKS / SKILLS / AGENTS / COMMANDS copy loops run. Without the re-exec, the in-memory Python interpreter is still executing the OLD `install.py` with the OLD `FEATURE_INCLUDES` / `SAME_PATH_FILES` constants — every new entry added in the upstream release (e.g. a new feature directory, a new skill, a new agent) is silently skipped because the in-memory `dict` doesn't know about it. Re-exec semantics:
    - **Re-exec target**: the freshly-copied `<target>/install.py` (NOT the source-side path under `<src>/install.py` and NOT the running script's `__file__`, since `__file__` is the OLD pre-update copy).
    - **Re-exec args**: the same `sys.argv` (so the new process sees the same `--update`, `--target`, `--version`, `--channel`, etc. flags). The `--src` arg, if present, is preserved verbatim.
    - **Infinite-loop guard**: BEFORE the `os.execv` call, set the env var `RABBIT_INSTALL_REEXEC_DONE=1` in `os.environ` (which `os.execv` inherits). At the very top of `main()`, check `os.environ.get("RABBIT_INSTALL_REEXEC_DONE") == "1"`; if set, skip the re-exec branch (the new process is already running the new code; one re-exec per `--update` invocation is enough). The new process MAY unset / leave the marker; it does not need to clear it because the parent process is the only re-exec source.
    - **Skip conditions** (no re-exec, the OLD code is correct / safe to continue with):
      (i) `--src` was explicitly supplied — the dev-test path and the install.sh first-install path; in both cases the in-memory code matches the source-of-truth content, so no version skew is possible.
      (ii) `RABBIT_INSTALL_REEXEC_DONE=1` is already set (the loop-guard above).
      (iii) The freshly-copied `<target>/install.py` is byte-identical to the running `__file__` (no upstream change — re-exec would be a no-op; skipping saves a fork).
    - **Ordering**: the re-exec MUST happen AFTER the `<target>/install.py` write completes (so the new code is on disk) and BEFORE any FEATURE_INCLUDES / HOOKS / SKILLS / AGENTS / COMMANDS copy loop runs (so those loops see the NEW dicts).
    - **stderr line on re-exec**: print a single visible line `re-execing into <target>/install.py with updated closure` to stderr before the `os.execv`, so the user sees the transition in the install log.
    Enforced by three tests under `.claude/features/rabbit-cage/test/`: `test-install-update-self-reexec.py` (re-exec into new code uses the new dict), `test-install-update-no-reexec-with-explicit-src.py` (skip-condition (i)), `test-install-update-reexec-loop-guard.py` (skip-condition (ii)).

    (g) **Self-fetch when invoked without `--src`.** When `--update` is set AND `--src` is omitted, `install.py main()` self-acquires the source by downloading the upstream tarball via stdlib `urllib.request` (NOT `curl`) into a `tempfile.TemporaryDirectory`, extracting via stdlib `tarfile`, locating the `rabbit-workflow-*` extracted dir, and treating it as the effective `--src`. The fetch URL is `https://github.com/{RABBIT_REPO}/archive/{REF}.tar.gz` with `RABBIT_REPO` defaulting to `changyu87/rabbit-workflow` and `REF` resolved per Inv 27 (CLI flag > env var > hardcoded stable release default; NEVER bleeding-edge `dev` unless explicitly opted in). When `--update` is set AND `--target` is omitted, `--target` is inferred as `Path(__file__).resolve().parent` (the directory containing the running install.py). Inferred-target sanity check: refuse with exit 1 and a clear stderr line when the inferred dir does NOT contain both `.claude/` and `.version` — this prevents accidental scaffolding into a non-rabbit-install directory. Fetch + extract MUST complete fully into the tempdir BEFORE any closure-overwrite begins, so network failure mid-fetch leaves the target byte-untouched. Network or extract failure (`URLError`, `OSError`, `tarfile.ReadError`, etc.) exits 1 with a single stderr line naming the URL and root cause; no retries. Explicit `--src` short-circuits the fetch (preserves the dev-test path and the install.sh path which both supply `--src`). Stays stdlib-only per Inv 4 Tech Stack. When fetching, if `RABBIT_INSTALLED_REF` env var is unset, set it to `RABBIT_REF` before `write_version_pin` runs (mirrors install.sh wiring per Inv 22e). Enforced by five tests under `.claude/features/rabbit-cage/test/`: `test-install-update-self-fetches-without-src.py` (mocked fetcher), `test-install-update-infers-target-from-script-location.py`, `test-install-update-rejects-non-rabbit-target.py` (refusal when inferred target lacks `.claude/`+`.version`), `test-install-update-network-failure.py` (URLError -> exit 1, target untouched), `test-install-update-explicit-src-skips-fetch.py` (pin the existing dev-test path).

25. **Session-override marker path-equality across all five consumers.** The session-override marker `.rabbit-scope-override` (content `'session'`) MUST be written, read, deleted, and alert-checked at a SINGLE per-mode canonical location. In plugin mode that location is `<repo_root>/.rabbit/.rabbit-scope-override` (where `<repo_root>` is the user-project root containing `.rabbit/.runtime/mode == 'plugin'`); in standalone mode it is `<repo_root>/.rabbit-scope-override`. Consumer (5) requires TWO `check_marker_alert` entries per event: one with relative path `'.rabbit-scope-override'` (resolves to the STANDALONE location) and one with relative path `'.rabbit/.rabbit-scope-override'` (resolves to the PLUGIN location). 26. **`install.sh` default ref resolves the LATEST published release DYNAMICALLY; the hardcoded value is an offline fallback only.** When the user supplies no explicit `RABBIT_REF` env-var override, `install.sh` MUST resolve the ref to fetch by querying GitHub's latest published release at install time — the SAME release source (`https://api.github.com/repos/<RABBIT_REPO>/releases/latest`, the `tag_name` field) that the contract-side update-check (`fetch_upstream_version` in `check-release-update.py`, contract Inv 63) and `install.py --update` use — so a fresh `curl … install.sh | bash` lands on the current latest release with NO per-release manual bump. The release process therefore never has to edit `install.sh` to advance the channel; first-install and update both track latest from the single GitHub release source.

    Resolution order (highest wins):
    - **`RABBIT_REF` env var** — when set, used verbatim (any branch, tag, or SHA, including `release/X.Y`, the main-centric default development tip `main`, or the legacy opt-in `dev`). The explicit override is fully supported and short-circuits the dynamic lookup; only the DEFAULT is dynamic. `main` and `dev` coexist as named branch refs during the transition to a main-centric default.
    - **Dynamic latest-release lookup** — when `RABBIT_REF` is unset, `install.sh` fetches `releases/latest` via `curl` (already a hard dependency of the script), parses the `tag_name` (a `vX.Y.Z` semver tag), and uses it as the fetch ref. The literal value `dev` is NEVER selected by the default path — only the published latest release tag.
    - **Offline fallback** — when the latest-release lookup fails (network unreachable, rate-limit / API outage, non-200, unparseable body, or empty `tag_name`), `install.sh` MUST degrade GRACEFULLY: it falls back to a hardcoded last-known-good release tag (the `RABBIT_FALLBACK_REF="vX.Y.Z"` literal near the top of the script, matching `v[0-9]+\.[0-9]+\.[0-9]+`) and prints a single clear stderr line naming the fallback. It MUST NOT silently install nothing and MUST NOT fall back to `dev`.

    The dynamic lookup deliberately adds a bootstrap-time network call to the latest-release endpoint. This is acceptable because: (a) `install.sh` already requires `curl` and already performs a network fetch for the tarball, so no NEW tooling dependency is introduced (`gh`/auth are NOT used — plain `curl` against the public API); (b) the failure mode is bounded by the offline fallback above; (c) it eliminates the chronic class of bug where a hardcoded default freezes installs many majors behind the actual latest release. The hardcoded `RABBIT_FALLBACK_REF` is a safety net, NOT the primary channel — it does not need a per-release bump to stay correct, though release cutters MAY advance it opportunistically.

    Enforced by `test/test-install-sh-resolves-latest-release.py`: parses `install.sh` and asserts (i) the default path performs a `releases/latest` API lookup (the endpoint substring is present); (ii) an explicit `RABBIT_REF` short-circuits the dynamic lookup (override honored verbatim); (iii) when the latest lookup fails, the script falls back to the `RABBIT_FALLBACK_REF` semver-tag literal and emits a clear stderr message (graceful offline degradation); (iv) the fallback literal matches `v[0-9]+\.[0-9]+\.[0-9]+` and is NOT `dev`. End-to-end via stubbed `curl` (a PATH-shadowed `curl` that serves a mocked `releases/latest` JSON body): the script selects the mocked latest tag, honors an explicit `RABBIT_REF`, and on a failing latest-lookup `curl` falls back to the hardcoded ref — all asserted against the ref actually fetched (the archive URL `curl` is called with).

27. **`install.py --update` ref resolution: CLI flag, env var, dynamic latest-release default (offline-fallback to a hardcoded tag) — NEVER silent `dev`.** `install.py main()` MUST resolve the upstream fetch ref under `--update` (Inv 22g) from a precedence ladder, with NO silent fallback to `dev`. The accepted inputs and their precedence (highest wins):

    (a) **`--version <ref>` / `--ref <ref>` CLI flag** — explicit argparse flag accepting any branch, tag, or SHA (e.g. `--version release/1.3`, `--ref v1.0.0`, `--ref dev`); both spellings are aliases. When supplied, this value is used verbatim and shell-agnostic (works in bash/zsh/csh/tcsh/fish without inline env-var syntax — the portable interface for shells that cannot inline-set `RABBIT_REF=…`).

    (b) **`--channel <name>` CLI flag** — convenience flag that expands to a ref. The named development channel is main-centric: it accepts three values — `main` (resolves to ref `main`, the default development tip), `dev` (resolves to ref `dev`, the legacy bleeding-edge channel retained as an explicit opt-in during the transition window), and `stable` (resolves via the dynamic latest-release default in (d)). The flag defaults to `stable` when omitted; explicit `--channel main` is the supported path for tracking the default development tip and explicit `--channel dev` is the supported opt-in for the legacy channel. The user MUST NOT land on `dev` (or any branch tip) without naming it. `main` and `dev` coexist so the named channel keeps working across the move to a main-centric default.

    (c) **`RABBIT_REF` env var** — fallback for scripted contexts (existing path; unchanged semantics from Inv 22g pre-amendment). Honored when neither `--version`/`--ref` nor `--channel main`/`--channel dev` is supplied.

    (d) **Dynamic latest-release default, with hardcoded offline fallback** — when none of (a)–(c) supplies a ref, `install.py` resolves the ref by querying GitHub's latest published release at fetch time, REUSING `fetch_upstream_version` from `.claude/features/contract/scripts/check-release-update.py` (the same logic the update-check and `/rabbit-update check` already use; cross-feature INVOKE declared in `contract.md invokes.scripts`). The returned `tag_name` (a `vX.Y.Z` semver tag) becomes the fetch ref, so `install.py --update` (and any default-ref path) tracks the latest release with no per-release manual bump. When the latest-release lookup fails (network/offline, non-200, unparseable, empty `tag_name`), `install.py` degrades GRACEFULLY to a hardcoded last-known-good semver-tag literal `HARDCODED_STABLE_DEFAULT` (matching `v[0-9]+\.[0-9]+\.[0-9]+`) and emits a single clear stderr line. The literal value `dev` is FORBIDDEN both as the resolved latest tag and as the fallback. The fallback is a safety net, NOT the primary channel; it does not require a per-release bump to stay correct.

    Effective ref selection (pseudo-code, applied at the start of the `--src is None` branch of `main()`):
    ```
    if args.version is not None: ref = args.version
    elif args.ref is not None:   ref = args.ref       # alias
    elif args.channel == "main": ref = "main"         # main-centric default tip

    ```

    The bootstrap-time network call to the latest-release endpoint is acceptable for the same reasons given in Inv 26: no NEW tooling dependency (stdlib `urllib` only, no `gh`/auth), a bounded failure mode (the offline fallback above), and elimination of the chronic stale-default bug.     The `--update` fetch path is ref-shape-agnostic: `fetch_upstream` builds the GitHub archive URL `https://github.com/<repo>/archive/<ref>.tar.gz`, which is the identical endpoint for a branch, a `vX.Y.Z` tag, or a SHA. The tags + GitHub Releases release model therefore needs NO change to the `--update` fetch mechanism — `install.py --update --version vX.Y.Z` already downloads the tag tarball; the default path now resolves that `vX.Y.Z` tag dynamically rather than from a frozen literal.

    **Downgrade guard (dynamic-default channel only).** On the dynamic-default channel — branch (d), when none of (a)–(c) supplies an explicit ref — `install.py --update` MUST NOT resolve to a ref OLDER than what is currently installed; the update ACTION tracks the update-CHECK and never goes BACKWARDS. When EITHER side carries no parseable semver, the comparison is indeterminate and the guard does NOT block (it never refuses an upgrade it merely failed to parse). The guard governs ONLY the dynamic-default channel: an explicit `--version`/`--ref`, `--channel dev`, or `RABBIT_REF` is honored verbatim and bypasses the guard, so an operator-named ref — even an intentional downgrade — still installs. When the marker is absent, the standalone and plugin decision trees behave exactly as before.
    Enforced by `test/test-scope-guard-decompose-context.py` (e2e via the real `scope-guard.py` subprocess): (i) marker present authorizing features A + B → a Write inside A's dir and inside B's dir are both ALLOWED with no per-feature marker; (ii) marker present authorizing A only → a Write inside an unauthorized feature C's dir is DENIED; (iii) marker absent → a Write inside A's dir with no per-feature marker is DENIED (normal bounded-scope rules); (iv) marker present but `expires` in the past → treated as absent (Write DENIED); (v) malformed-JSON or empty-`features` marker → treated as absent (Write DENIED); (vi) regression — a per-feature `.rabbit-scope-active-<feature>` marker and a legacy `session` override still ALLOW as before with the decompose marker absent. Wired into `test/run.py`. `hooks/scope-guard.py` is a deployed hook (`publish_hook` in this feature's MANIFEST); its deployed copy under `.claude/hooks/` drifts until republished.

48. **A local `--src` install with no resolvable published ref pins a MEANINGFUL local marker into `.version`, never the bare sentinel.** When `install.py main()` writes `<target>/.version`, the pin label is taken from the `RABBIT_INSTALLED_REF` env var when it is set and non-empty (the published-install and `--update` self-fetch path, which always sets it). When that env var is unset or empty — the local `--src` install case (`install.py --src <local checkout> --target <proj>/.rabbit`) — `install.py` MUST derive a meaningful local marker rather than pinning the literal string `unknown`. The `.version` content flows verbatim into BOTH the SessionStart version box (Inv via `_read_installed_version` → `rabbit v<content>`) and the contract update-check headline (`current: <content> ... on channel <content>`), so a sentinel pin renders `rabbit vunknown` and `current: unknown ... on channel unknown`, comparing a real upstream release against a non-version. The derived marker is `local-<short-sha>` obtained from a read-only `git -C <src_root> rev-parse --short HEAD` against the `--src` source tree, falling back to the literal `local` when no SHA is resolvable (git absent, the source is not a git checkout). The derivation is exposed as the `install.py` helper `write_version_pin(dst_root, src_root=None)` (and its private `_local_src_marker(src_root)`), is best-effort and NEVER raises (a git failure degrades to `local`). Enforced by `test/test-install-version-pin-local-src-not-unknown.py`: (i) a git-checkout `--src` with no `RABBIT_INSTALLED_REF` pins a label carrying the source short SHA and NOT equal to `unknown`; (ii) a non-git `--src` falls back to `local`, still not `unknown`; (iii) an explicit `RABBIT_INSTALLED_REF` still wins verbatim; (iv) end-to-end — the produced `.version`, fed through the deployed SessionStart dispatcher, yields a banner containing NEITHER `vunknown` NOR `channel unknown`. Wired into `test/run.py`.

49. **The `--update` path TOLERATES a closure that SHRINKS across a surface removal; a fresh install still HARD-FAILS on any dangling source.** The `--update` path is driven by the LOCALLY-installed (older) `install.py`, whose hardcoded closure list still names surfaces the NEW upstream source has intentionally REMOVED (deleted or renamed files). The fresh-install integrity self-check (Inv 21) treats EVERY closure source absent from `--src` as a hard "dangling required-file" abort; applied verbatim on `--update` it makes a release that crosses a surface removal permanently un-installable from an older install, because the stale local closure can never be reconciled before the re-exec into the NEW `install.py` (Inv 22h). To keep the primary supported update path working across surface removals, the integrity gate MUST be asymmetric by mode:
    (a) **Fresh install (no `--update`).** Behavior is unchanged: `main()` runs `check_install_sources_exist(src_root)` and HARD-ABORTS (exit 1, naming every offending path) when ANY closure source is absent from `--src`. A fresh install's running `install.py` and its closure are self-consistent with the source it ships from, so any missing source is a genuine packaging defect.
    (b) **Update (`--update`).** A closure source the NEW `--src` does NOT contain is a TOLERATED closure SHRINK — a surface the upstream release dropped — and is DROPPED rather than treated as a dangling abort: `main()` MUST NOT fail the update on these old-closure-only absences. The dropped entries are simply skipped by the per-file copy loops (`copy_one` no longer aborts the run for a source the update mode tolerates), so the in-place refresh proceeds to the Inv 22h re-exec, after which the NEW `install.py` validates its OWN (corrected) closure against the NEW source. Concretely, on `--update` the per-closure-list copy loops (`SAME_PATH_FILES`, `HOOKS`, `SKILLS`, `AGENTS`, `COMMANDS`, `FEATURE_INCLUDES`) skip any `(src_rel, dst_rel)` whose `src_rel` is absent from `src_root` instead of recording a hard failure, and the pre-copy integrity self-check does not abort on the shrink.
    (c) **A genuine dangling ref stays a HARD failure even on `--update`.** Tolerance is scoped EXACTLY to the SHRINK: an old-closure entry absent from the new source. It is NOT a license to skip arbitrary missing files. The NEW `install.py` post-re-exec carries the NEW closure; a path the NEW closure REQUIRES but the NEW source does NOT contain (a real packaging defect in the upstream release, not a surface removal) is caught by the same Inv 21 self-check running under the NEW code and HARD-FAILS the install, and the same path is screened at CI by the fresh-install integrity tests and the cross-feature contract gate (`check_install_sources_exist` against the real repo). The integrity check is therefore NOT weakened for real dangling refs — only the old-installer's view of a removed surface is reconciled.
    The exported helper `check_install_sources_exist(repo_root)` is unchanged (still the deterministic, importable enumeration of every absent closure source); the asymmetry lives in how `main()` CONSUMES it — abort on fresh install, tolerate-and-drop on `--update`. Enforced by `test/test-install-update-tolerates-closure-shrink.py` (e2e via the real `install.main()`): (i) an `--update` whose stale local closure names a source the NEW `--src` has dropped SUCCEEDS (exit 0) and the dropped entry is skipped, not copied; (ii) a fresh install (no `--update`) against a source missing a required closure source still HARD-ABORTS (exit 1) naming the offending path; (iii) an `--update` whose NEW closure requires a source absent from the NEW source still HARD-FAILS (over-tolerance guard). Wired into `test/run.py`.

50. **Every rabbit-cage vendored-mode value comparison DUAL-ACCEPTS both `"vendored"` and `"plugin"`.** The canonical vendored-mode value — the string `detect_mode` returns and `write_mode_marker` writes into `<repo_root>/.rabbit/.runtime/mode` — is owned by rabbit-meta, NOT by rabbit-cage. While that canonical value is being renamed from `"plugin"` to `"vendored"`, rabbit-cage's consumers MUST compare against BOTH spellings via the membership test `mode in ("vendored", "plugin")` so the gate stays green both BEFORE the rename (value still `"plugin"`) and AFTER it (value `"vendored"`). The dual-accept is a deliberate COEXISTENCE shim, not the long-term shape. Every rabbit-cage site that branches on the vendored-mode value MUST dual-accept: `scripts/show-mode.py` (the `detect_mode` return value governing the project-root derivation, Inv 45), `hooks/scope-guard.py` (the two `<repo_root>/.rabbit/.runtime/mode` content reads — the vendored-branch dispatch in `decide()` and the per-mode override-marker path in `_override_marker_path()`), and `scripts/scope-guard-on.py` (its mirror of `_override_marker_path`). rabbit-cage NEVER WRITES the marker value (that is `write_mode_marker`, a contract API) and NEVER changes `detect_mode`; it only READS/COMPARES, so dual-accept is the complete rabbit-cage-side preparation. **The dual-accept obligation extends to rabbit-cage's OWN test assertions that OBSERVE the vendored-mode value**: any rabbit-cage test that drives the real `write_mode_marker` (which writes `detect_mode`'s value VERBATIM) or the real `detect_mode`/`show-mode.py` output against a vendored-signature layout MUST assert the observed marker content / `mode` field / human summary with the SAME dual-accept membership test (`value in ("vendored", "plugin")`, or "EITHER spelling appears" for the human summary line) — never a strict `== "plugin"` / `!= "plugin"` / `"plugin" not in` pin — so the suite stays green both before and after the rename. A test that pins the OBSERVED vendored-mode value strictly to `"plugin"` would itself go RED on the `detect_mode → "vendored"` flip even though the code under test already dual-accepts. The dual-accept test-assertion sites are: `test/test-show-mode-command.py` (the t1 `mode` field check and the t3 human-summary check; t5 already dual-accepts), `test/test-write-mode-marker-wired.py` (the marker-content check after a real SessionStart dispatch in a vendored-signature layout), and `test/test-mode-marker-root-consistency.py` (the canonical-path marker-content check in a faithful vendored install layout). Test fixtures that WRITE a chosen `"plugin"` marker as an INPUT to exercise a consumer (e.g. the `test-plugin-*` scope-guard/alert tests) are unaffected — they pin their own valid legacy input, not an observed `detect_mode` output. Enforced by `test/test-mode-value-dual-accept.py` (e2e): drives `scope-guard.py` and `scope-guard-on.py` as subprocesses against vendored layouts whose `.runtime/mode` marker holds `"plugin"` in one run and `"vendored"` in another, asserting the vendored-mode override-marker path (`<repo_root>/.rabbit/.rabbit-scope-override`) and the vendored decision branch fire identically for BOTH values; asserts `show-mode.py` derives the parent-`project_root` for both values; and GUARDS the test-assertion-side dual-accept by driving the real `contract.lib.runtime.write_mode_marker` against a vendored layout with `detect_mode` stubbed to return each of `"plugin"` and `"vendored"` and confirming the written marker content satisfies the dual-accept membership test (and equals the stubbed value verbatim), demonstrating the observe-side assertion accepts BOTH spellings. Wired into `test/run.py`. **Coexistence deprecation criterion:** the `"plugin"` alternative is removed from every dual-accept site — collapsing the membership test back to a single `mode == "vendored"` comparison — once rabbit-meta's `detect_mode`/`write_mode_marker` have flipped the canonical value to `"vendored"` AND no installed `.rabbit/.runtime/mode` marker in a supported install can still contain the legacy `"plugin"` string (i.e. after the next `--update` cadence rewrites every live marker).
